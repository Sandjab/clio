# CLIO v0.13 â€” RESCUE `step.error` + `RESUME` Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close two v0.8 ergonomic gaps on the RESCUE handler â€” expose `step.error.message` and `step.error.type` as dotted kwarg values inside rescue bodies, and add `RESUME(<step>.<field>)` as a second legal terminator next to `abort("...")`.

**Architecture:** Extend the existing parser â†’ IR â†’ emitter pipeline. No new modules. New AST/IR nodes (`ErrorAccessExpr` / `ErrorAccessIR`, `ResumeAst` / `ResumeIR`). `_parse_call_arg` accepts 3-segment dotted kwarg values when the middle token is the literal `error`. The RESCUE body's union widens to accept `ResumeIR` as a legal terminator alongside the current `CallIR("abort", ...)`. Python and MCP emitters add a `_err` parameter to the helper signature, emit `str(_err)` / `type(_err).__name__` substitutions in helper bodies, and dispatch the wrapper squelette on the terminator type at emit time (no runtime sentinel).

**Tech Stack:** Python 3.12, frozen dataclasses, pytest. No new dependencies. See the validated design at `docs/superpowers/specs/2026-05-14-clio-v0.13-rescue-error-resume-design.md` for the full rationale and edge cases.

                self.advance()  # consume DOT

    def _parse_resume_call(self) -> "ResumeAst":
        """`RESUME ( IDENT . IDENT )` â€” terminator of a rescue chain.
        Caller has already peeked the RESUME keyword."""
        kw_tok = self.expect(TokenType.KEYWORD, "RESUME")
        self.expect(TokenType.LPAREN)
        step_tok = self.peek()
        if step_tok.type != TokenType.IDENT:
            raise ParseError(
                f"RESUME requires '<step>.<field>', got {step_tok.type.value} {step_tok.value!r}",
                kw_tok.line, kw_tok.col,
            )
        self.advance()
        self.expect(TokenType.DOT)
        field_tok = self.expect(TokenType.IDENT)
        # Allow exactly one (step.field) pair: refuse extra args / extra dots.
        if self.peek().type == TokenType.DOT:
            raise ParseError(
                "RESUME accepts exactly '<step>.<field>'; no further dotted path",
                kw_tok.line, kw_tok.col,
            )
        if self.peek().type == TokenType.COMMA:
            raise ParseError(
                "RESUME takes a single '<step>.<field>' argument; remove the comma",
                kw_tok.line, kw_tok.col,
            )
        self.expect(TokenType.RPAREN)
        return ResumeAst(
            fallback_step=step_tok.value,
            field_name=field_tok.value,
            line=kw_tok.line,
            col=kw_tok.col,
        )
```

```markdown
## Fallback via RESUME â€” recover from a judgment step failure

**Goal:** when a judgment step fails after ON_FAIL exhausts, run a rule-based fallback and inject its result so the flow continues normally.

**Pattern:**

```clio
STEP detect
  TAKES:   rows: List<int>
  GIVES:   report: churn_report
  MODE:    judgment
  ON_FAIL: retry(3) then escalate

STEP fallback_detect
  TAKES: rows: List<int>
  GIVES: report: churn_report
  MODE:  exact

STEP notify
  TAKES: channel: str, reason: str, err_type: str
  GIVES: sent: bool
  MODE:  exact

FLOW pipeline
  load(path="data.csv")
    -> detect(rows=rows)
    -> route_alerts(report=report)

  RESCUE detect:
    -> notify(channel="#alerts",
              reason=detect.error.message,
              err_type=detect.error.type)
    -> fallback_detect(rows=rows)
    -> RESUME(fallback_detect.report)
```

**What this does** at runtime:

1. `load` runs.
2. `detect` runs. If it raises, ON_FAIL's `retry(3)` and then `escalate` execute first.
3. If those exhaust, the RESCUE body runs:
   - `notify` posts to Slack â€” `detect.error.message` is the string of the captured exception, `detect.error.type` is its class name.
   - `fallback_detect` runs a rule-based version of the analysis.
   - `RESUME(fallback_detect.report)` replaces `state["report"]` with `fallback_detect`'s report.
4. The flow continues normally: `route_alerts(report=report)` reads the injected fallback value.

The compiler validates at build time that `fallback_detect.report`'s type matches `detect.report`'s type. They must be structurally equal.

**When to prefer this** over `abort("...")`: when a deterministic fallback gives an acceptable answer (rules, cached previous result, default) and the rest of the flow can usefully run on top of it.
```

- [ ] **Step 2: Troubleshooting entries**

Append to `docs/manual/06-troubleshooting.md`, one entry per compile-time error:

```markdown
### `<step>.error.<field> can only reference the step protected by this RESCUE`

**Cause:** A `step.error.message` (or `.type`) kwarg value refers to a step that is not the one protected by the enclosing RESCUE handler.

**Fix:** rewrite the reference to match the rescued step:
```clio
RESCUE detect:
  -> notify(reason=detect.error.message)    # OK
  -> notify(reason=load.error.message)      # ERROR â€” load is not the rescued step
```

### `unknown error field 'X', expected one of ['message', 'type']`

**Cause:** Only `step.error.message` (str) and `step.error.type` (str = Python exception classname) are supported in v0.13. Other fields (`.stacktrace`, `.cause`, `.context`) are out of scope.

**Fix:** use one of the two supported fields. If you need additional context, capture it into the `notify` step's body (e.g., have your `notify` impl read `traceback.format_exc()` itself).

### `<step>.error.<field> is only valid inside a RESCUE handler`

**Cause:** `step.error.message` / `.type` was used as a kwarg value outside any RESCUE body â€” e.g., in the main FLOW chain.

**Fix:** error access has no meaning outside a RESCUE body. Move the reference into a RESCUE handler attached to the step you want to inspect.

### `RESUME(<step>.<field>): step '<step>' is not called in this RESCUE handler`

**Cause:** `RESUME(X.Y)` references a step `X` that does not appear earlier in the same RESCUE chain.

**Fix:** call `X` as a step in the handler body before the RESUME line:
```clio
RESCUE detect:
  -> fallback(rows=rows)              # <-- declare the fallback call
  -> RESUME(fallback.report)          # <-- then RESUME from it
```

### `RESUME(<step>.<field>): '<field>' is not a field of step '<step>'`'s GIVES'

**Cause:** The field named in RESUME does not exist in the fallback step's `GIVES`.

**Fix:** check the GIVES of the fallback step and use one of its declared fields.

### `RESUME(<step>.<field>): type T1 is incompatible with rescued step's GIVES type T2`

**Cause:** The fallback field's type doesn't structurally match the rescued step's GIVES type. v0.13 requires strict equality.

**Fix:** either change the fallback step's GIVES so the types align, or wrap/transform the value via an intermediate `exact` step whose GIVES matches the rescued step.

### `RESCUE <step>: body must end with abort(...) or RESUME(...)`

**Cause:** The handler chain doesn't end with either `abort("...")` or `RESUME(<step>.<field>)`. Every RESCUE must terminate with exactly one of these.

**Fix:** add the missing terminator. If the handler purely side-effects and you don't want to recover, end with `abort("...")` with a contextual message.
```
