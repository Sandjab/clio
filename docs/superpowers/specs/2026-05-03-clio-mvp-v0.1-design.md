# CLIO Compiler MVP v0.1 — Design

**Date** : 2026-05-03
**Status** : approved by user, ready for writing-plans
**Scope** : first runnable slice of the CLIO compiler, target `claude-cli` only

## 1. Goals & non-goals

### Goal

Build a Python compiler that takes a `.clio` source file (restricted to the v0.1 subset defined in §3) and emits a Claude Code project that runs end-to-end against the real `claude -p` CLI.

### Concrete success criterion

Running:

```bash
python -m clio compile examples/mvp.clio --target claude-cli --output ./output
bash ./output/run.sh
```

must:

1. produce the emitted project structure described in §6 with files matching the golden fixtures byte-for-byte;
2. read `customers.csv` (a placeholder shipped with the output);
3. invoke `claude -p --model haiku` once for the `detect_churn` judgment step;
4. validate the LLM response against the JSON Schema derived from `CONTRACT customer_risk`;
5. write the validated `risks` array into `state.json`;
6. exit 0.

### Non-goals (deliberately deferred)

The parser must produce a clear `not yet supported in v0.1` error (not silent ignore) when any of these appear:

- `MODE: auto`, `LANG: auto` (mode/lang inference)
- Step fields: `CACHE`, `VALIDATE`, `ON_FAIL`
- Contract field: `CONFIDENCE`
- Control flow: `FOR EACH`, `WHILE`, `IF` / `ELSE`, `MATCH` / `CASE` / `DEFAULT`
- Resources fields: `budget`, `prefer`, `strategy`
- Targets other than `claude-cli`

Also out of scope: the optimizer (batching, model routing, context budgeting), Claude Code hooks-based validation, the natural-language → `.clio` frontend (Phase 2).

## 2. Test policy

CI runs golden file tests only (no API calls).

The end-to-end test that actually invokes `claude -p` is gated by `CLIO_E2E=1` and is run manually before declaring v0.1 done. This avoids API key plumbing in CI but ensures the emitted project is verified to actually run.

## 3. v0.1 language subset

The MVP example below is the literal grammar coverage. The parser supports exactly what is needed to parse it (plus the explicit error cases for non-goals).

```
CONTRACT customer_risk
  SHAPE:    {client: str, risk: enum(low|mid|high), reason: str(max=300)}
  ASSERT:   len(reason) > 0

STEP load_customers
  TAKES:    file: CSV
  GIVES:    customers: List<{name: str, revenue: float}>
  MODE:     exact

STEP detect_churn
  TAKES:    customers: List<{name: str, revenue: float}>
  GIVES:    risks: List<customer_risk>
  MODE:     judgment

FLOW customer_retention
  load_customers(file="customers.csv")
    -> detect_churn(customers)

RESOURCES
  target:   claude-cli
  models:   [haiku]
```

Induced grammar coverage:

- **Declarations**: `CONTRACT`, `STEP`, `FLOW`, `RESOURCES`
- **Step fields (v0.1)**: `TAKES`, `GIVES`, `MODE` (`exact` | `judgment` only)
- **Contract fields (v0.1)**: `SHAPE`, `ASSERT`
- **Resources fields (v0.1)**: `target`, `models`
- **Types**: primitives (`int`, `float`, `str`, `bool`), `List<T>`, records `{f: T, ...}`, enums `enum(a|b|c)`, constrained `str(max=N)`, domain `CSV`, contract refs by name
- **Flow constructs**: sequential chain `->`, step calls with kwargs (`name(arg=value)`)
- **ASSERT expression sublanguage** (§5)
- **Lexical**: `#` comments (line and inline), Python-style indentation for blocks, double-quoted strings, integer and float literals

## 4. Pipeline

```
.clio source
  → Lexer            (parser/lexer.py)            token stream
  → Parser           (parser/parser.py)           AST (frozen dataclasses)
  → IR Builder       (ir/builder.py)              FlowGraph (frozen, type-checked, contract-resolved)
  → Optimizer        (ir/optimizer.py)            no-op v0.1 (identity), kept for future hookability
  → ClaudeCLIEmitter (emitters/claude_cli.py)     output directory
```

The compiler is a pure function: same source → same output bytes. No timestamps, no random ordering, no embedded paths from the host machine.

## 5. ASSERT expression sublanguage

A small, closed expression grammar. Only what is needed for v0.1:

```
expr   := comparison
comparison := term op term
op     := == | != | >= | <= | > | <
term   := IDENT
        | NUMBER
        | STRING
        | call
call   := IDENT '(' expr (',' expr)* ')'
```

Allowed builtins in `call`: `len` only (other names are an error).
Allowed identifiers in `term`: only field names of the contract being defined.

Compilation: parsed into an AST node, serialized as a Python source string (e.g. `"len(reason) > 0"`), embedded in the emitted JSON Schema under `x-clio-assert`. The runtime helper `clio_runtime.validate` reads it and `eval`s it in a restricted namespace that contains only `len` and the field values. No imports, no attribute access, no other builtins. Restriction is enforced by passing `{"__builtins__": {}}` as globals to `eval` and the explicit allow-list as locals.

## 6. Emitted project structure

```
output/
  CLAUDE.md                       human-readable project description, models, refs to source .clio
  .claude/hooks.json              {} (placeholder for v0.2)
  steps/
    01_load_customers.py          Python script: parse CSV, write customers to state.json
    02_detect_churn.prompt        template with ${customers} placeholder + schema instruction footer
    02_detect_churn.schema.json   JSON Schema for List<customer_risk> (uses $ref into contracts/)
  contracts/
    customer_risk.schema.json     standalone schema for the contract (with x-clio-assert)
  clio_runtime/
    __init__.py
    validate.py                   copied verbatim from compiler's clio/runtime/validate.py
    substitute.py                 copied verbatim; safe ${var} substitution from state.json
  state.json                      initial: {} ; populated by run.sh as steps execute
  customers.csv                   placeholder example data (5 rows, the user can replace)
  run.sh                          bash orchestrator
  README.md                       one paragraph: "bash run.sh"
```

Key choice: the `clio_runtime/` module is **copied** into the emitted project, not pip-installed. The emitted project must be self-contained (only `python` and `claude` required at runtime, plus `jq` and `jsonschema` available on PATH or installed).

## 7. Orchestrator (`run.sh`) — emitted shape

```bash
#!/usr/bin/env bash
set -euo pipefail
cd "$(dirname "$0")"
echo '{}' > state.json

# Step 1: load_customers (exact, python)
python steps/01_load_customers.py --file customers.csv

# Step 2: detect_churn (judgment)
PROMPT=$(python -m clio_runtime.substitute steps/02_detect_churn.prompt state.json)
RESPONSE=$(printf '%s' "$PROMPT" | claude -p --model haiku --output-format text)
printf '%s' "$RESPONSE" | python -m clio_runtime.validate steps/02_detect_churn.schema.json -
jq --argjson r "$RESPONSE" '.risks = $r' state.json > state.json.tmp && mv state.json.tmp state.json

echo "[clio] flow customer_retention completed. See state.json."
```

Variable substitution is delegated to `clio_runtime.substitute` (a small Python helper) rather than `envsubst`, because JSON values containing quotes, newlines, or shell metacharacters do not survive bash variable interpolation reliably.

## 8. Contract validation — emitted schema example

`CONTRACT customer_risk` with `SHAPE: {client: str, risk: enum(low|mid|high), reason: str(max=300)}` and `ASSERT: len(reason) > 0` compiles to:

```json
{
  "type": "object",
  "properties": {
    "client": {"type": "string"},
    "risk":   {"enum": ["low", "mid", "high"]},
    "reason": {"type": "string", "maxLength": 300}
  },
  "required": ["client", "risk", "reason"],
  "additionalProperties": false,
  "x-clio-assert": "len(reason) > 0"
}
```

The judgment step's schema (`02_detect_churn.schema.json`) wraps this:

```json
{
  "type": "array",
  "items": {"$ref": "../contracts/customer_risk.schema.json"}
}
```

`clio_runtime.validate` runs `jsonschema.validate(instance, schema)` with `RefResolver` rooted at the project dir, then if `x-clio-assert` is present on `items` (or on the schema itself for non-array contracts), evaluates the expression for each item with the field values bound as locals.

## 9. CLI surface

```
python -m clio compile <source.clio> --target claude-cli --output <dir>
python -m clio check   <source.clio>
```

Exit codes:
- `0` — success
- `1` — parse error, type error, or validation error (with line:col message on stderr)
- `2` — I/O error (file not found, output dir not writable)

No other commands in v0.1.

## 10. Error reporting

Every AST node carries `line: int` and `col: int`. Lexer, parser, and IR errors all reference the source location. Standard format:

```
mvp.clio:14:7: type mismatch in flow customer_retention
  detect_churn expects: customers: List<{name: str, revenue: float}>
  load_customers gives: customers: List<{name: str, revenue: float, last_order: str}>
```

For "not yet supported" features (the explicit non-goals listed in §1), the format is:

```
mvp.clio:9:3: 'CACHE' is not supported in CLIO v0.1 (planned for a later milestone)
```

## 11. Compiler structure

```
clio/
  __init__.py
  __main__.py               python -m clio entrypoint
  cli.py                    argparse: compile / check
  keywords.py               closed enum, single source of all keywords
  parser/
    __init__.py
    lexer.py
    parser.py
    ast_nodes.py            frozen dataclasses, line/col on every node
    expressions.py          mini parser for ASSERT (§5 grammar)
  ir/
    __init__.py
    builder.py              AST → FlowGraph
    graph.py                FlowGraph, StepIR, ContractIR (frozen)
    types.py                type system + structural type checking
    contracts.py            SHAPE → JSON Schema, ASSERT → Python source string
    optimizer.py            no-op v0.1 (placeholder)
  emitters/
    __init__.py
    base.py                 abstract BaseEmitter
    claude_cli.py           ClaudeCLIEmitter
  runtime/                  copied into emitted projects (see note below)
    __init__.py
    validate.py
    substitute.py
tests/
  fixtures/
    mvp.clio                the v0.1 north star
    expected/mvp/           golden output tree
    invalid/                .clio files + .expected_error pairs
  test_lexer.py
  test_parser.py
  test_ir.py
  test_types.py
  test_emitters/test_claude_cli.py    # diff every emitted file vs golden
  test_e2e.py                          # gated by CLIO_E2E=1
```

All AST and IR nodes are frozen dataclasses (Pydantic v2 models for the contract schema generation only). Immutability is enforced because optimizer passes are expected later, and we want to diff stages cleanly.

**Runtime package rename**: the compiler's `clio/runtime/` package is copied into emitted projects under the top-level name `clio_runtime/` (not as a sub-package of `clio`). Reason: the emitted project must work standalone, without the compiler installed; a top-level name avoids any chance of collision if both happen to be on the same `sys.path`.

## 12. Test strategy

| Layer | What is tested | How |
|-------|----------------|-----|
| Lexer | token types and positions | unit tests on small source snippets |
| Parser | AST structure for valid programs; error message + line/col for invalid ones | corpus of valid `.clio` + `tests/fixtures/invalid/*.clio` paired with `*.expected_error` files |
| IR builder | type-checking, contract resolution, "not yet supported" rejections | unit tests on AST → FlowGraph, plus invalid corpus |
| Emitter | byte-exact output | compile `mvp.clio`, diff every file vs `tests/fixtures/expected/mvp/`. Mismatch fails with an actionable diff. |
| End-to-end | emitted project actually runs | `test_e2e.py`, gated by `CLIO_E2E=1`. Compiles `mvp.clio`, runs `bash output/run.sh`, asserts `state.json.risks` is a non-empty array of objects each conforming to the contract schema. |

## 13. Implementation order — vertical slices

Each slice delivers a working pipeline (parser → IR → emitter) for a slightly larger language subset, with green tests, before moving on.

| # | Slice | Test added at this slice |
|---|-------|--------------------------|
| 1 | Skeleton: empty `STEP foo` declaration only | compile produces a directory with stub files, doesn't crash |
| 2 | Step fields with primitive types only (`TAKES`, `GIVES`, `MODE`) | echo-back step in Python |
| 3 | Type expressions: `List<T>`, records, enums | JSON Schema fragments emitted correctly |
| 4 | `CONTRACT` with `SHAPE` only + `ContractRef` resolution | schema.json emitted for the judgment step |
| 5 | Constrained types: `str(max=N)` | constraints land in JSON Schema as `maxLength` |
| 6 | `FLOW` + sequential chain `->` + inter-step type checking | run.sh with two real steps |
| 7 | Judgment runtime: `.prompt` template + `${var}` substitution + `claude -p` invocation + `clio_runtime.validate` | substitution and validation paths |
| 8 | `ASSERT` mini-expression parser → embedded `x-clio-assert` | runtime fails fast on invariant violation |
| 9 | `RESOURCES` (`target`, `models`) | `--model haiku` injected into run.sh |
| 10 | End-to-end on full `mvp.clio` with `CLIO_E2E=1` | real claude -p call passes |

Each slice ends with a commit. Slices 1–9 use only golden file tests. Slice 10 adds the gated e2e test.

## 14. Open points (acknowledged but not blocking v0.1)

- **`.claude/hooks.json` shape for v0.2**: when we add `CACHE`, `ON_FAIL`, etc., we will reconsider whether validation moves into hooks. v0.1 keeps it in `run.sh` deliberately.
- **Multi-step kwargs binding**: the v0.1 example only chains `step_a -> step_b(arg)` where `arg` is a name from `step_a`'s `GIVES`. Anything more complex (literal args, subscripting) is not supported and will error.
- **Schema `$ref` resolution at runtime**: `jsonschema` needs a `RefResolver`. The runtime helper sets it up rooted at the output directory. If we later emit to a different layout, this is the seam to revisit.

## 15. What this design rules out

- **No half-built features**: we never emit a partially-supported `CACHE` or `ON_FAIL`. They are parser-rejected with a clear message.
- **No silent ignores**: unsupported fields raise errors, not warnings.
- **No runtime dependencies on the compiler**: the emitted project does not import `clio.*` — only the copied `clio_runtime/` module.
- **No host-specific output**: paths in emitted files are relative; no embedded usernames, timestamps, or absolute paths.
