# CLIO v0.3 — Python emitter — Design

**Date** : 2026-05-03
**Status** : draft, awaiting user approval before plan
**Scope** : add a second emitter (`target: python`) that produces a runnable Python package (Anthropic SDK + Pydantic) from the same IR. Validates the "IR is target-independent" architecture claim. Builds on v0.2 (`v0.2.0` tag).

## 1. Goals & non-goals

### Goal

Compile any v0.2 `.clio` source to a Python package with the same semantics as the `claude-cli` target:
- exact steps become user-implemented Python functions;
- judgment steps become full implementations that call the Anthropic SDK, validate against Pydantic models, honor `CACHE`, and run the `ON_FAIL` strategy chain;
- contracts become Pydantic v2 models;
- the FLOW becomes an orchestrator function;
- a CLI entry point lets users run the flow from the shell.

### Concrete success criterion

`examples/mvp.clio` (already at v0.2 form: CACHE + ON_FAIL chain + fallback) compiles to **both** `--target claude-cli` and `--target python`. The Python output:

1. Installs cleanly with `pip install -e .` from the emitted directory.
2. Exposes both an importable API (`from retention.flow import run; result = run(file="customers.csv")`) and a CLI (`python -m retention --file customers.csv`).
3. After the user fills in the exact step bodies (`load_customers`, `detect_churn_naive`), runs end-to-end against real `claude -p` (gated under `CLIO_E2E=1`).
4. Cache hit on second run within TTL: zero Anthropic API calls (verified via SDK monkeypatch in the test).
5. ON_FAIL strategy chain behaves identically to claude-cli: retry on validation failure, escalate to next model with cache key recompute, fallback to the named exact step, abort with the configured message.
6. The compiler is still a pure function: same source → byte-identical Python output.

### Non-goals (deferred to v0.4+)

- **Async / concurrent step execution** — sync only in v0.3.
- **Provider-neutral SDK** (httpx + raw HTTP) — `anthropic` SDK only. A future emitter could be `python-openai` or `python-bedrock` if needed.
- **Streaming** — full-response only. Streaming would change the strategy chain shape (partial responses can't be retry-recovered cleanly).
- **Tool use / function calling** — judgment steps emit JSON via prompt + schema (same as claude-cli). Tool use is a future feature.
- **Pluggable LLM backend** at runtime (e.g. swap haiku→openai-gpt4 via env var) — the model list in `RESOURCES.models` is baked into the emitted code.
- **Auto-generated unit tests** for the emitted package — users write their own. The compiler emits a stub `tests/` directory with one smoke test.
- **State persistence to disk by default** — state lives in a Python dict. Optional `--state-file` argument for parity with claude-cli's `state.json` (out of v0.3 scope; can land in v0.3.x).

The compiler explicitly rejects each unsupported feature with a clear "not yet supported in target python" error rather than silently emitting broken code.

## 2. Test policy

The CI suite (no API calls) gains:
- Golden-file tests for the Python emitter (per-file content asserted byte-identical, like the existing `claude-cli` goldens).
- Pydantic model validation tests (round-trip a sample dict through the emitted models).
- Strategy chain unit tests via SDK monkeypatch (no network): emit, import, monkeypatch the `Anthropic` client to return predetermined responses, assert retry/escalate/fallback fire correctly.

A new e2e test, gated by `CLIO_E2E=1`, mirrors the slice-I cache-replay test:
1. Compile `examples/mvp.clio --target python --output ./out_py`.
2. `pip install -e ./out_py` into a temporary venv (or use `.venv` directly).
3. Run once against real `claude -p` → populates cache, validates output.
4. Reset state, re-import the package, monkeypatch `Anthropic.messages.create` to record calls; second `run()` must produce zero recorded API calls (cache hit).

## 3. Language additions

**None.** v0.3 keeps the same `.clio` grammar as v0.2. The Python emitter consumes the existing IR.

The only compiler-surface change is the `--target` choice in `clio/cli.py` (now `{claude-cli, python}` instead of `{claude-cli}`).

## 4. Output structure

For an input flow named `customer_retention`, `python -m clio compile --target python --output ./out` produces:

```
out/
  pyproject.toml
  README.md
  customer_retention/
    __init__.py
    contracts.py                # Pydantic models from CONTRACT declarations
    flow.py                     # orchestrator function `run(**kwargs)`
    __main__.py                 # CLI entry point (argparse + run())
    steps/
      __init__.py
      load_customers.py         # exact step (user-implemented)
      detect_churn_naive.py     # exact step (user-implemented)
      detect_churn.py           # judgment step (full implementation)
    clio_runtime/
      __init__.py
      cache.py                  # copied verbatim from clio/runtime/cache.py
```

The package name is derived from the `FLOW` name (snake_case, lowercase). If the FLOW name is `customer_retention`, the package is `customer_retention`. If it conflicts with a stdlib name (rare), the user can rename post-hoc.

`pyproject.toml`:

    "anthropic>=0.40",

## 5. Pydantic model emission

For each `CONTRACT name SHAPE: {field: type, ...}`:

```python
# contracts.py
from typing import Literal
from pydantic import BaseModel, Field, field_validator


class CustomerRisk(BaseModel):
    """CONTRACT customer_risk
    SHAPE: {client: str, risk: enum(low|mid|high), reason: str(max=300)}
    ASSERT: len(reason) > 0
    """
    client: str
    risk: Literal["low", "mid", "high"]
    reason: str = Field(max_length=300)

    @field_validator("reason")
    @classmethod
    def _assert_reason_nonempty(cls, v: str) -> str:
        if not (len(v) > 0):
            raise ValueError("ASSERT failed: len(reason) > 0")
        return v
```

Type mapping:
- `int` → `int`
- `float` → `float`
- `str` → `str`
- `bool` → `bool`
- `enum(a|b|c)` → `Literal["a", "b", "c"]`
- `str(max=N)` → `str = Field(max_length=N)`
- `List<T>` → `list[T_python]` (recurse on T)
- record `{f: T, ...}` (anonymous) → nested `BaseModel` with auto-generated name (`{Parent}_F` if not at top-level), or inline `dict[str, ...]` for shallow cases. **Decision**: always emit a nested BaseModel — keeps validation behavior consistent and enables IDE autocompletion.
- contract ref by name → reference to the named class.

`ASSERT` expressions translate to a `@field_validator` (or `@model_validator(mode="after")` if multi-field). v0.3 supports the v0.1 `ASSERT` mini-grammar (`len(x) > N`, `==/!=/>=/<=/>/<` with int/float/str literals). The translation is direct since the AST is already structured (see `clio/parser/expressions.py`).

## 6. Step emission

### Exact step

```python
# steps/load_customers.py
"""STEP load_customers (exact)
TAKES: file: str
GIVES: customers: list[dict]  # [{name: str, revenue: float}]

Implement the body below. The orchestrator passes `file` as a keyword
argument and expects a list of dicts conforming to the GIVES type.
"""


def load_customers(*, file: str) -> list[dict]:
    raise NotImplementedError(
        "Implement steps/load_customers.py: parse the CSV at `file` "
        "and return a list of {name, revenue} dicts."
    )
```

The body is a `NotImplementedError` stub. Users replace it. The compiler does NOT generate the implementation for exact steps — they are by definition deterministic Python the user owns.

For the v0.3 example (`examples/mvp.clio`), the README includes a copy-paste body for `load_customers` (CSV parsing) and `detect_churn_naive` (the heuristic), mirroring the v0.2 README's instructions.

### Judgment step

```python
# steps/detect_churn.py
"""STEP detect_churn (judgment)
CACHE:   ttl(86400)
ON_FAIL: retry(3) then escalate then fallback(detect_churn_naive) then abort("...")
"""
import json
import os
import sys
from pathlib import Path

from anthropic import Anthropic

from .. import contracts
from ..clio_runtime import cache as _cache
from . import detect_churn_naive as _detect_churn_naive


_PROMPT_TEMPLATE = """\
You receive a list of customers as JSON. Output a JSON array of risk
assessments...

Customers:
${customers}

Schema:
${schema}
"""

_INLINED_SCHEMA = '{"type":"array","items":{"type":"object","properties":{...},"required":[...]}}'

_MODELS = ("claude-haiku-4-5-20251001", "claude-sonnet-4-6", "claude-opus-4-7")
_CACHE_TTL = 86400


def _attempt(model: str, prompt: str) -> str | None:
    """Returns the raw response string on validation success, None on failure."""
    try:
        client = Anthropic()
        msg = client.messages.create(
            model=model,
            max_tokens=4096,
            messages=[{"role": "user", "content": prompt}],
        )
        raw = msg.content[0].text if msg.content else ""
        if not raw:
            return None
        # Strip markdown fences (mirror the bash awk filter).
        cleaned = "\n".join(line for line in raw.splitlines() if not line.startswith("```"))
        # Validate via Pydantic.
        parsed = [contracts.CustomerRisk.model_validate(item) for item in json.loads(cleaned)]
        # Re-serialize through the validated models for consistency.
        return json.dumps([p.model_dump() for p in parsed])
    except Exception:
        return None


def detect_churn(*, customers: list[dict]) -> list[contracts.CustomerRisk]:
    prompt = _PROMPT_TEMPLATE
    prompt = prompt.replace("${customers}", json.dumps(customers))
    prompt = prompt.replace("${schema}", _INLINED_SCHEMA)

    cache_dir = Path(os.environ.get("CLIO_CACHE_DIR", ".cache"))

    # Cache lookup (primary model)
    primary_key = _cache.cache_key("detect_churn", _MODELS[0], prompt, _INLINED_SCHEMA)
    hit = _cache.cache_lookup(cache_dir, "detect_churn", primary_key, _CACHE_TTL)
    if hit is not None:
        return [contracts.CustomerRisk.model_validate(r) for r in json.loads(hit)]

    model_idx = 0
    fallback_used = False
    response: str | None = None

    # Initial attempt
    response = _attempt(_MODELS[model_idx], prompt)

    # retry(3)
    if response is None:
        for _ in range(3):
            response = _attempt(_MODELS[model_idx], prompt)
            if response is not None:
                break

    # escalate
    if response is None and model_idx < len(_MODELS) - 1:
        model_idx += 1
        esc_key = _cache.cache_key("detect_churn", _MODELS[model_idx], prompt, _INLINED_SCHEMA)
        esc_hit = _cache.cache_lookup(cache_dir, "detect_churn", esc_key, _CACHE_TTL)
        if esc_hit is not None:
            return [contracts.CustomerRisk.model_validate(r) for r in json.loads(esc_hit)]
        response = _attempt(_MODELS[model_idx], prompt)
        if response is not None:
            _cache.cache_store(cache_dir, "detect_churn", esc_key, _MODELS[model_idx], response)

    # fallback(detect_churn_naive)
    if response is None:
        fallback_response = _detect_churn_naive.detect_churn_naive(customers=customers)
        # Re-serialize through Pydantic for consistency
        validated = [contracts.CustomerRisk.model_validate(r) for r in fallback_response]
        response = json.dumps([v.model_dump() for v in validated])
        fallback_used = True

    # abort
    if response is None:
        print("[clio] step detect_churn: churn detection exhausted", file=sys.stderr)
        raise SystemExit(1)

    # Cache store under primary key (only if primary model succeeded AND no fallback)
    if model_idx == 0 and response is not None and not fallback_used:
        _cache.cache_store(cache_dir, "detect_churn", primary_key, _MODELS[0], response)

    return [contracts.CustomerRisk.model_validate(r) for r in json.loads(response)]
```

This is verbose by design — every strategy is observable, no hidden orchestration. Mirrors the bash run.sh structure 1:1.

## 7. Orchestrator + CLI

```python
# flow.py
"""FLOW customer_retention
load_customers(file=...) -> detect_churn(customers)
"""
from . import contracts
from .steps import detect_churn, load_customers


def run(*, file: str = "customers.csv") -> dict:
    customers = load_customers.load_customers(file=file)
    risks = detect_churn.detect_churn(customers=customers)
    return {
        "customers": customers,
        "risks": [r.model_dump() for r in risks],
    }
```

```python
# __main__.py
"""CLI entry point. `python -m customer_retention --file customers.csv`."""
import argparse
import json
import sys

from .flow import run


def main(argv: list[str] | None = None) -> int:
    parser = argparse.ArgumentParser(prog="customer_retention")
    parser.add_argument("--file", default="customers.csv")
    args = parser.parse_args(argv)
    result = run(file=args.file)
    json.dump(result, sys.stdout, indent=2, default=str)
    sys.stdout.write("\n")
    return 0


if __name__ == "__main__":
    raise SystemExit(main())
```

## 8. Cache + ON_FAIL — interaction (Python form)

Same semantics as v0.2 §6 (claude-cli). The Python form is a direct transcription:

- Cache lookup at entry (against primary model's key).
- On miss: initial attempt → retry loop → escalate (with new-model cache key recompute) → fallback (sets `fallback_used=True`) → abort.
- Cache store only when (a) MODEL_IDX is still 0 (primary), (b) response is non-None, (c) `fallback_used` is False.
- Each model's own successful response is cached under its own key (escalate path stores under `esc_key`).

Implementation is ~80 lines per judgment step. Less concise than the bash version because Python lacks bash's empty-string-as-falsy idiom, but more debuggable.

## 10. Implementation order — vertical slices

Each slice ends with a green test suite and a commit. Following the v0.2 pattern.

| # | Slice | Adds |
|---|-------|------|
| A | CLI extension + `PythonEmitter` skeleton | `--target python` accepted; emitter writes pyproject.toml + empty package directories. Two tests: smoke compile + golden of pyproject.toml. |
| B | Pydantic model emission | `contracts.py` generated from CONTRACTs. Type mapping for primitives, enum, str(max=N), List, nested records, contract refs. ASSERT → field validator. Goldens for 2-3 contract shapes. |
| C | Exact step stubs | `steps/<name>.py` files with type-hinted signatures, NotImplementedError body, docstring derived from the IR. Goldens. |
| D | Judgment step body — single attempt, no cache, no on_fail | Full Python body that calls Anthropic SDK, validates via Pydantic, returns the response. No cache, no strategy chain yet. Goldens + monkeypatched smoke test. |
| E | CACHE wrapping in judgment steps | Copy `cache.py` into emitted package. Lookup at entry, store on success. Same TTL semantics as bash. Golden + monkeypatch test for cache hit. |
| F | ON_FAIL strategy chain | retry(N), escalate (with cache recompute), fallback (with FALLBACK_USED tracking), abort. SDK monkeypatch tests for each strategy. |
| G | Orchestrator + CLI entry | `flow.py` (calls steps in chain order), `__main__.py` (argparse + run). Smoke test imports the package and calls `run()` with monkeypatched SDK. |
| H | Example + e2e gated test | Compile `examples/mvp.clio --target python` into a fixture. `tests/test_e2e_python.py` gated by `CLIO_E2E=1`: pip install, real claude run, cache replay verification. README of emitted package documents the user-edit steps. |
| I | Documentation + release v0.3.0 | CHANGELOG v0.3.0, README "Current status" mentions both targets, docs/COMPILATION_TARGETS.md gains a "python" section, docs/ARCHITECTURE.md unchanged (the diagram already shows "Emitter" generically). Tag v0.3.0. |

Commit per slice. Each slice's tests must be green before the next.

## 11. Error reporting

The Python emitter does NOT introduce new compile-time errors — it consumes the same validated IR as claude-cli. Any IRBuildError raised before emission is identical for both targets. The only target-specific surface is the `--target python` choice handler in `cli.py`.

At runtime in the emitted Python:
- Empty / malformed LLM responses → caught by `_attempt`, returned as None, triggers strategy chain.
- Pydantic validation failures → caught by `_attempt`, returned as None.
- All strategies exhausted → `SystemExit(1)` with `[clio] step <name>: <abort_message>` on stderr (or implicit `[clio] step <name>: ON_FAIL strategies exhausted`).
- Anthropic SDK errors (auth, network, rate limit) → propagate as the SDK's typed exceptions. The strategy chain treats them as failures (caught in the `except Exception:` in `_attempt`).

## 13. Open points (acknowledged, not blocking v0.3)

- **Async / parallel execution** — sync only in v0.3. v0.4 could add `async def` variants if the FLOW gains parallel branches (`FOR EACH ... PARALLEL`).
- **State persistence** — the dict returned by `run()` is the state. Optional disk persistence (parity with claude-cli's `state.json`) is a v0.3.x feature.
- **Multiple FLOWs** in one source — out of scope for v0.3 (the parser already rejects this in `build_ir`). When supported, the Python package would have one orchestrator function per flow.
- **Provider-neutral SDK** — out of scope. A `python-openai` emitter would be a fork that targets a different SDK, not a runtime config.
- **Anthropic prompt caching** for system prompts — supported by the SDK but not exercised in v0.3 because v0.2 doesn't have a notion of "system" vs "user" prompt (just the rendered template). Add when prompt sections are formalized.

## 14. What this design rules out

- **No silent caching of unvalidated responses** — same as v0.2. A response that fails Pydantic validation is never written to cache.
- **No silent fallthrough on unknown ON_FAIL strategies** — handled at IR build time (parser is the single source of truth).
- **No cross-step cache key collision** — step name is part of the key.
- **No automatic retry backoff** — same as v0.2; retries are immediate.
- **No reliance on a Claude Code installation** — the emitted Python project depends on `anthropic` SDK + `pydantic` only. No bash, no jq, no claude CLI.
