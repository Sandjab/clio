# CLIO v0.13 — `step.error` in RESCUE + `RESUME` terminator — design

**Date**: 2026-05-14
**Sprint**: v0.13
**Status**: Spec drafted, awaiting user review before writing the implementation plan.

## Motivation

v0.8 shipped the `RESCUE` handler but left two ergonomic holes that LANGUAGE_SPEC §RESCUE explicitly flagged for v0.9+:

1. The handler body cannot inspect the captured error — `step.error.X` is documented as reserved but the parser rejects it. Handlers can notify Slack but cannot say *what* failed.
2. The handler must always terminate in `abort(...)`. There is no path to "recover and continue" — even when a `fallback_<step>` exists in scope and could supply a valid GIVES.

v0.13 closes both gaps in one sprint:

- `rescued_step.error.message` (str) and `rescued_step.error.type` (str = Python exception classname) become valid expressions, **only** inside the RESCUE body of the rescued step.
- A new `RESUME(<step>.<field>)` terminator joins `abort(...)` as a legal end of the rescue chain. RESUME injects a value type-compatible with the rescued step's GIVES, and the flow continues normally from the next item after the rescued step.

The sprint touches only the `python` and `mcp-server` emitters. `claude-cli` and `langgraph` continue to reject RESCUE at compile time (v0.8 rule unchanged).

## Decisions made during brainstorm

| Topic | Decision | Rejected alternatives |
|---|---|---|
| Error access granularity | Typed fields: `.message` (str), `.type` (str = Python exception classname) | (a) string-only `step.error` — loses type info; (b) `.message`/`.type`/`.stacktrace` — stacktrace too Python-specific and leaks paths; (c) dict opaque — breaks compile-time typing |
| RESUME in scope? | **Included** in v0.13 | Deferred to a later sprint — would have left v0.13 too narrow given the open spec promise |
| RESUME form | `RESUME(<step>.<field>)` referencing a step called earlier in the handler chain | (a) `RESUME(<literal>)` — compile-time type check ambiguous for non-trivial GIVES; (b) both forms accepted — twice the validation surface, no concrete user need |
| Error access scope | Strict: `X.error.Y` only valid where X is the step rescued by the enclosing RESCUE handler | Cross-step access — breaks RESCUE's "one handler per failure" mental model |
| Implementation strategy | Option 1: extend the IR with `ErrorAccessIR` and `ResumeIR` nodes, validate at IR build time, dispatch terminator squelette at emit time | Option 2 (string-magic in emitters) — violates "source line in every error" project rule; Option 3 (error as a Record) — too invasive for the scope |
| Top-level RESCUE limitation | **Kept** — the rescued step must still appear in the top-level FLOW chain | Lifting it — separate sprint (touches IR scope tracking) |

## IR shape

### New nodes (`clio/ir/graph.py`)

```python
@dataclass(frozen=True)
class ErrorAccessIR:
    """rescued_step.error.field reference.

    Only valid inside a RESCUE body. `rescued_step` MUST equal the
    step protected by the enclosing RESCUE handler. `field` MUST be
    one of {"message", "type"}.
    """
    rescued_step: str
    field: str
    source_line: int


@dataclass(frozen=True)
class ResumeIR:
    """RESUME(<step>.<field>) terminator.

    `fallback_step` MUST be a step called earlier in the same RESCUE
    handler chain. `field_name` MUST be a key in fallback_step's GIVES.
    The type of that field MUST match the GIVES type of the step
    rescued by this handler.
    """
    fallback_step: str
    field_name: str
    source_line: int
```

### Changes to `RescueBlockIR`

```python
@dataclass(frozen=True)
class RescueBlockIR:
    step_name: str
    chain: tuple[FlowItemIR, ...]    # unchanged
    terminator: AbortIR | ResumeIR   # NEW — union; was "implicit last item must be AbortIR"
    source_line: int
```

The previous rule "last top-level item must be `abort(...)`" widens to "must be `abort(...)` OR `RESUME(...)`". Backward-compatible: existing flows produce a `RescueBlockIR` with `terminator: AbortIR` and no other code changes.

### Validation rules (enforced at IR build time, with source line)

| Rule | Error message |
|---|---|
| `ErrorAccessIR.rescued_step` ≠ enclosing RESCUE's `step_name` | `line N: X.error.Y can only reference the step protected by this RESCUE (got X, expected Z)` |
| `ErrorAccessIR.field` ∉ {message, type} | `line N: unknown error field 'F', expected one of ['message', 'type']` |
| `ErrorAccessIR` outside a RESCUE body | `line N: step.error.X is only valid inside a RESCUE handler` |
| `ResumeIR.fallback_step` not in current handler chain | `line N: RESUME(X.Y): step 'X' is not called in this RESCUE handler` |
| `ResumeIR.field_name` not in `fallback_step.GIVES` | `line N: RESUME(X.Y): 'Y' is not a field of step X's GIVES (got: [a, b, c])` |
| `ResumeIR.field_name` type ≠ rescued_step GIVES type (strict structural equality on the IR `Type`; v0.13 supports only strict match) | `line N: RESUME(X.Y): type T1 is incompatible with rescued step's GIVES type T2` |
| Handler chain has no terminator | `line N: RESCUE body must end with abort(...) or RESUME(...)` |
| ON_FAIL ends in abort AND RESCUE present (any terminator) | `line N: step X: ON_FAIL ends in abort(...); RESCUE handler is unreachable — remove one` |

The first 7 are sprint-new. The last is the existing v0.8 rule generalised: previously it specifically referenced "redundant double-abort"; v0.13 generalises to "unreachable handler" since RESCUE may now terminate in RESUME.

## Parser & lexer

### New keyword

```python
# clio/keywords.py — additions
RESUME = "RESUME"   # uppercase, consistent with RESCUE / IF / WHILE
```

`error`, `message`, `type` remain plain `IDENT` tokens. Their reservation is contextual (rejected at IR build time, not at lex time).

### Grammar update

LANGUAGE_SPEC §RESCUE grammar widens to:

```
rescue_block      := "RESCUE" step_name ":" NEWLINE INDENT
                       rescue_chain
                     DEDENT

rescue_chain      := (flow_item ("->" flow_item)* "->")? rescue_terminator
rescue_terminator := abort_call | resume_call
abort_call        := "abort" "(" STRING ")"                # unchanged
resume_call       := "RESUME" "(" IDENT "." IDENT ")"      # new

value_expr        := ... | error_access
error_access      := IDENT "." "error" "." IDENT           # new — kwarg value, strict 3-segment shape
```

### Parsing strategy

1. **Error access** is detected by extending `_parse_call_arg` (the function that parses kwarg values in step calls). Today it accepts STRING, NUMBER, or a single IDENT (state-ref shorthand returned as `"@<name>"`). We extend it to also accept a 3-segment dotted path `IDENT "." IDENT "." IDENT` whose middle token is the literal `error`:
   - `name=value` → existing literal kwarg behaviour (STRING / NUMBER)
   - `name=ident` → existing state-ref shorthand (`"@ident"`)
   - `name=step.error.field` → new `ErrorAccessAst(step, field, line)`
   - `name=step.field` (2 segments) → currently disallowed in kwargs; remains disallowed in v0.13 (separate sprint if ever needed)
   - `name=a.b.c` with `b != "error"` → ParseError with message `unknown 3-segment kwarg value 'a.b.c'; only <step>.error.<message|type> is supported as a dotted kwarg value`

2. **`resume_call`** is parsed by `parse_rescue_chain`. After each flow_item, the parser peeks:
   - `RESUME` keyword → parse `RESUME ( IDENT . IDENT )`, build `ResumeAst`, ensure no `->` follows
   - `abort` IDENT → existing path, build `AbortAst`
   - else → ParseError `RESCUE body must end with abort(...) or RESUME(...)`

3. **Strict shape for RESUME**: `RESUME(step)` (no field), `RESUME(literal)`, `RESUME()`, `RESUME(step.field, ...)`, all raise ParseError with explicit messages pointing to the source line.

### New AST nodes (`clio/parser/ast_nodes.py`)

```python
@dataclass(frozen=True)
class ErrorAccessAst:
    step_name: str
    field: str            # parsed as-is; IR build validates {message, type}
    line: int

@dataclass(frozen=True)
class ResumeAst:
    fallback_step: str
    field_name: str
    line: int
```

`RescueBlockAst.chain` reads `flow_item` items; the last entry of the parsed structure is the terminator (an `AbortAst | ResumeAst`), stored separately from the chain's `flow_item` tuple.

## Emitter — `python`

### Pattern with abort terminator (with `.error` substitutions)

```python
def _rescue_detect_churn(state, _err):
    state["sent"] = notify_slack(channel="#alerts", reason=str(_err))
    raise FlowAborted(f"churn detection failed - {type(_err).__name__}")

# In flow.py:
try:
    state["report"] = detect_churn(rows=state["rows"])
except FlowAborted:
    raise
except Exception as _err:
    _rescue_detect_churn(state, _err)
    raise  # unreachable; handler raised
```

Differences from v0.12:

- Helper signature: `_rescue_<step>(state)` → `_rescue_<step>(state, _err)`
- Wrapper binds the exception: `except Exception as _err:`
- `detect_churn.error.message` substitution: `str(_err)`
- `detect_churn.error.type` substitution: `type(_err).__name__`

### Pattern with RESUME terminator

```python
def _rescue_detect_churn(state, _err):
    state["sent"] = notify_slack(channel="#alerts", reason=str(_err))
    state["fallback_result"] = fallback_churn(rows=state["rows"])
    return state["fallback_result"]   # RESUME(fallback_churn.fallback_result)

# In flow.py:
try:
    state["report"] = detect_churn(rows=state["rows"])
except FlowAborted:
    raise
except Exception as _err:
    state["report"] = _rescue_detect_churn(state, _err)
```

Helper `return`s instead of `raise`. Wrapper assigns the helper's return value to the **rescued step's** state slot (`state["report"]` — the GIVES of `detect_churn`, not of `fallback_churn`). The flow continues normally on the next line, which reads `state["report"]`.

### Squelette dispatch

The emitter inspects `rescue.terminator`:

- `AbortIR` → abort squelette (helper raises, wrapper `raise` after call)
- `ResumeIR` → resume squelette (helper returns, wrapper assigns)

No runtime sentinel, no `isinstance` check at runtime. Compile-time chooses the squelette → zero overhead, zero magic.

### Cross-target invariant

`class FlowAborted(Exception)` remains module-local in `flow.py`, still gated on `rescues` being non-empty → flows without RESCUE produce **byte-identical** output to v0.12.

Flows with RESCUE but neither `.error` nor RESUME also produce byte-identical output to v0.12 (helper signature gains `_err` but the body has no substitutions, so the only diff is the unused parameter — to be confirmed in the regression snapshot test).

**Decision pending in implementation**: do we always emit `_err` in the helper signature for the abort case, or only when the body actually references `step.error.X`? Trade-off: always-on keeps the signature stable across recompiles; only-when-used keeps v0.12 byte-identical output for RESCUE+abort with no error access.

→ **Resolution**: always emit `_err` in the signature, but use `_err` (single underscore prefix) so Python lints / Ruff don't flag it as unused. Output is no longer byte-identical to v0.12 for RESCUE+abort flows, but the diff is minimal (one parameter added) and the cross-target invariant for "flows without RESCUE = byte-identical to v0.12" is preserved.

## Emitter — `mcp-server`

Strict async mirror of the Python emitter.

### Abort case

```python
async def _rescue_detect_churn(state, _err, _session):
    state["sent"] = await notify_slack(channel="#alerts", reason=str(_err), _session=_session)
    raise FlowAborted(f"churn detection failed - {type(_err).__name__}")

try:
    state["report"] = await detect_churn(rows=state["rows"], _session=_session)
except FlowAborted:
    raise
except Exception as _err:
    await _rescue_detect_churn(state, _err, _session)
    raise  # unreachable
```

### Resume case

```python
async def _rescue_detect_churn(state, _err, _session):
    state["sent"] = await notify_slack(channel="#alerts", reason=str(_err), _session=_session)
    state["fallback_result"] = await fallback_churn(rows=state["rows"], _session=_session)
    return state["fallback_result"]

try:
    state["report"] = await detect_churn(rows=state["rows"], _session=_session)
except FlowAborted:
    raise
except Exception as _err:
    state["report"] = await _rescue_detect_churn(state, _err, _session)
```

### Differences vs Python emitter

- `def` → `async def`
- All step calls inside the handler get `await` + `_session=_session`
- Wrapper site is `async`-aware (already the case in v0.8 MCP emitter)

### Shared helpers

The walker that emits `step.error.X` substitutions and the dispatch `terminator → squelette` both live in helpers shared between Python and MCP emitters. The two emitters differ only on `async`/`await` and `_session` threading — same pattern validated in v0.11 and v0.12.

## v0.13 limitations (carried forward into the manual)

- One RESCUE per step (v0.8 rule, unchanged).
- Protected step must be in the top-level FLOW chain (v0.8 rule, unchanged).
- No nested or recursive RESCUE. A handler raised exception propagates without being caught by another handler.
- `step.error.X` is restricted to 2 fields (`message`, `type`). No `.stacktrace`, no `.cause`, no `.context`.
- `RESUME` takes only `<step>.<field>`. No literals, no expressions, no inline record construction.
- `claude-cli` and `langgraph` continue to reject RESCUE entirely.
- If a step called inside the handler chain (`fallback_step` in `RESUME(fallback_step.field)`) itself raises, the exception propagates — there is no nested handler.

## Out of scope (deferred)

- `RESUME(literal)` form (e.g. `RESUME({risks: []})`)
- `step.error.stacktrace`, `step.error.cause`
- RESCUE on a non-top-level step (inside FOR EACH / IF / MATCH / WHILE body)
- RESCUE in LangGraph (planned for the next sprint in the option-2 sequence)
- RESCUE in claude-cli (no current plan — bash orchestrator semantics don't map cleanly)
- "Always-emit `_err`" vs "only-when-used" is decided in favour of always-emit; the alternative is documented above
