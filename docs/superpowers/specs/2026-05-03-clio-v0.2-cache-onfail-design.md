# CLIO v0.2 — CACHE + ON_FAIL — Design

**Date** : 2026-05-03
**Status** : approved by user, ready for writing-plans
**Scope** : add reproducibility (`CACHE`) and resilience (`ON_FAIL`) to judgment steps. Builds on v0.1 (`v0.1.0` tag).

## 1. Goals & non-goals

### Goal

Make CLIO flows **reproducible** (CACHE) and **resilient to LLM non-determinism** (ON_FAIL retry / escalate / fallback / abort), so the same `.clio` source can be developed iteratively without re-spending API tokens, and so transient model failures do not abort the whole flow.

### Concrete success criterion

The MVP example in `examples/mvp.clio`, augmented with `CACHE: ttl(24h)` and `ON_FAIL: retry(3) then escalate`, satisfies all of:

1. **First run** invokes `claude -p` once (haiku) and produces a validated `state.json.risks`.
2. **Second run within 24h, same input**: zero `claude -p` invocations, identical `state.json.risks` (cache hit, deterministic byte-for-byte).
3. **A run where the model returns a non-conformant JSON** is automatically retried up to 3 times on the same model; if still failing, the next model in `RESOURCES.models` is tried once.
4. **A run where every model fails the schema** invokes the configured `fallback(step_name)` step (an `exact` rule-based fallback) and uses its output.
5. **A run where everything fails** exits 1 with a clear message indicating which step exhausted its strategies.
6. The compiler is still a pure function (same source → byte-identical output).

### Non-goals (deferred to v0.3+)

- `ON_FAIL` on `MODE: exact` steps (exact-step failures still abort immediately).
- `CONFIDENCE` field on contracts.
- `VALIDATE` field on steps (distinct from `ASSERT` on contracts).
- Caching of `exact` steps (different concern: code determinism, not LLM reproducibility).
- `IF <step>.FAILS` control-flow keyword (an alternative routing mechanism that overlaps with ON_FAIL fallback; will be addressed when control flow lands).
- The optimizer (model routing heuristics, batching, context budgeting).

The parser explicitly rejects each of the above with a clear "not yet supported" error rather than ignoring.

## 2. Test policy

CI runs the unit + golden-file test suite (no API calls). The end-to-end test stays gated by `CLIO_E2E=1`. v0.2 adds a second e2e assertion: after a successful first run, deleting `state.json` and re-running within the TTL must complete **without** an `claude -p` invocation. This is verified by interposing a PATH-stub `claude` script during the second run that records calls — zero recorded calls means the cache short-circuited the LLM as required.

## 3. v0.2 language additions

The MVP example is extended to:

```
CONTRACT customer_risk
  SHAPE:  {client: str, risk: enum(low|mid|high), reason: str(max=300)}
  ASSERT: len(reason) > 0

STEP detect_churn
  TAKES:    customers: List<{name: str, revenue: float}>
  GIVES:    risks:     List<customer_risk>
  MODE:     judgment
  CACHE:    ttl(24h)
  ON_FAIL:  retry(3) then escalate then fallback(detect_churn_naive) then abort("churn detection exhausted")

STEP detect_churn_naive
  TAKES:    customers: List<{name: str, revenue: float}>
  GIVES:    risks:     List<customer_risk>
  MODE:     exact

STEP load_customers
  TAKES:    file:      CSV
  GIVES:    customers: List<{name: str, revenue: float}>
  MODE:     exact

FLOW customer_retention
  load_customers(file="customers.csv")
    -> detect_churn(customers)

RESOURCES
  target:  claude-cli
  models:  [haiku, sonnet, opus]
```

### New keywords

`CACHE`, `on`, `off`, `ttl`, `ON_FAIL`, `retry`, `then`, `escalate`, `fallback`, `abort`.

### New token type

`DURATION` — matches `\d+[smhd]` (e.g. `24h`, `7d`, `30s`, `5m`). Lexed as one token; the parser converts to seconds (`s=1`, `m=60`, `h=3600`, `d=86400`).

### Grammar

```
StepDecl       ::= STEP IDENT NEWLINE INDENT step_field+ DEDENT
step_field     ::= TAKES_clause | GIVES_clause | MODE_clause
                | CACHE_clause                            (NEW, judgment-only)
                | ON_FAIL_clause                          (NEW, judgment-only)

CACHE_clause   ::= "CACHE" ":" cache_value NEWLINE
cache_value    ::= "on"
                | "off"
                | "ttl" "(" DURATION ")"

ON_FAIL_clause ::= "ON_FAIL" ":" strategy_chain NEWLINE
strategy_chain ::= strategy ("then" strategy)*
strategy       ::= "retry" "(" NUMBER ")"
                | "escalate"
                | "fallback" "(" IDENT ")"
                | "abort" "(" STRING ")"
```

Parser rejects `CACHE` or `ON_FAIL` on a `MODE: exact` step with `<file>:<line>:<col>: 'CACHE' (or 'ON_FAIL') is only supported on judgment steps in v0.2`.

## 4. CACHE — semantics

### Modes

- `on` — permanent cache. Hit on any existing entry, regardless of age.
- `off` — default. No lookup, no write. Behaves exactly like v0.1.
- `ttl(<duration>)` — cache with expiration. Entry is a hit if `now - created_at < duration_seconds`. Expired entries are not deleted (they are overwritten on next successful run).

### Cache key

```
sha256("\n".join([step_name, model, rendered_prompt, inlined_schema_json]))
```

Hex-encoded, no truncation. The four inputs are separated by newlines (deterministic JSON or text rendering ensures stability):

- `step_name` — IR step name.
- `model` — the actual model used for this attempt (so a cache populated under `haiku` does not satisfy a request for `sonnet`).
- `rendered_prompt` — the prompt template after `${var}` substitution from `state.json` (i.e. exactly what `claude -p` will receive on stdin, *minus* the `${schema}` placeholder which is filled separately).
- `inlined_schema_json` — the inlined schema bash variable content (the same string that bash substitutes into the prompt). The on-disk `.schema.json` (with `$ref`) is NOT used here — we hash the form the LLM actually sees.

### Storage layout

In the emitted project:

```
output/
  .cache/
    <step_name>/
      <sha256_hex>.json
```

Override via env var `CLIO_CACHE_DIR=/some/path`. When set, the layout becomes `$CLIO_CACHE_DIR/<step_name>/<key>.json`.

### Entry format

```json
{
  "created_at": 1746284400,
  "model": "haiku",
  "response": "[{\"client\": \"Alpha Corp\", ...}, ...]"
}
```

`created_at` is Unix epoch seconds. `response` is the raw response string (post-markdown-strip), the same value `validate.py` accepted.

### Lookup → write rules

- Lookup happens **before** any `claude -p` invocation, against the key computed from the *first* model in the strategy (typically the one named in `RESOURCES.models[0]`, before any `escalate`).
- A successful response (post-validation) is written to cache only if the step has a non-`off` `CACHE` policy, and only if the response was produced — whether on the first attempt, a retry, or after an escalate. Fallback-step output is **not** written under the main step's cache key (the fallback step manages its own cache if any).
- Concurrent writers: the cache file is written atomically (write to `<key>.json.tmp`, `mv` into place).

### Module: `clio_runtime/cache.py`

Public surface:

```python
def cache_key(step_name: str, model: str, prompt: str, schema_json: str) -> str: ...

def cache_lookup(cache_dir: Path, step_name: str, key: str, ttl_seconds: int | None) -> str | None:
    """Return the cached response string if a fresh entry exists, else None.
    `ttl_seconds=None` means permanent (mode `on`)."""

def cache_store(cache_dir: Path, step_name: str, key: str, model: str, response: str) -> None:
    """Atomically write the cache entry. Creates parent dirs as needed."""

def main(argv): ...   # CLI: subcommands `key`, `lookup`, `store`
```

CLI usage from `run.sh`:

```bash
KEY="$("$PYTHON" -m clio_runtime.cache key "$STEP_NAME" "$MODEL" "$PROMPT" "$SCHEMA")"
HIT="$("$PYTHON" -m clio_runtime.cache lookup ".cache" "$STEP_NAME" "$KEY" 86400 || true)"  # exits 0 on hit (prints response), 1 on miss (silent)
"$PYTHON" -m clio_runtime.cache store ".cache" "$STEP_NAME" "$KEY" "$MODEL" "$RESPONSE"
```

(`lookup` exits 1 with no output on miss, exits 0 with the response on stdout on hit. Bash captures both with `|| true` and tests `[ -z "$HIT" ]`.)

## 5. ON_FAIL — semantics

### Strategies

| Strategy | Behavior on entry |
|----------|-------------------|
| `retry(N)` | Re-run the same step (re-render prompt from current state, re-call `claude -p`) up to N additional times — so up to N+1 total attempts when this is the first clause and the initial attempt is counted. A "failure" is: claude exit non-0, empty RAW response, validation failure, or assert failure. |
| `escalate` | Bump `model_idx` by 1 (i.e. move to the next entry in `RESOURCES.models`). Run **one** attempt with the new model. If `model_idx` is already at the last model, this clause is a no-op and the next clause is tried. |
| `fallback(step_name)` | Run the named step with the same TAKES values as the failing step. The fallback step runs its own ON_FAIL chain if defined. If the fallback step ultimately produces a validated GIVES, it counts as success and that GIVES becomes the main step's GIVES (substituted into state.json). If the fallback step itself exhausts its strategies, this clause fails and the next clause is tried. |
| `abort("<message>")` | Exit `run.sh` with status 1 and the message on stderr (prefixed with `[clio] step <name>:`). No further clauses are tried. |

### Composition

`ON_FAIL: <s1> then <s2> then <s3> ...` means: try strategies left-to-right, advancing on failure. The first clause's "initial attempt" is the original step run (counted in retry's N+1).

The chain composes into a runtime decision flow. The diagram below shows the canonical case `CACHE: ttl(...)` + `ON_FAIL: retry(N) then escalate then fallback(naive) then abort("...")`:

```mermaid
flowchart TD
    entry([STEP entry]) --> cache{cache lookup<br/>under primary key}
    cache -- "hit" --> apply([apply RESPONSE to state.json])
    cache -- "miss / off" --> initial["initial attempt<br/>_clio_run_attempt(model[0])"]
    initial --> ok1{validated?}
    ok1 -- "yes" --> store_primary["cache store<br/>under primary key"]
    store_primary --> apply
    ok1 -- "no" --> retry["retry(N): up to N more<br/>attempts on current model"]
    retry --> ok2{validated?}
    ok2 -- "yes" --> store_primary
    ok2 -- "no" --> escalate{next model<br/>available?}
    escalate -- "yes" --> bump["MODEL_IDX++<br/>recompute KEY_NN_ESC"]
    bump --> esc_lookup{cache hit on<br/>new model's key?}
    esc_lookup -- "yes" --> apply
    esc_lookup -- "no" --> esc_attempt["attempt on new model"]
    esc_attempt --> ok3{validated?}
    ok3 -- "yes" --> store_esc["cache store<br/>under KEY_NN_ESC"]
    store_esc --> apply
    ok3 -- "no" --> fallback["run fallback step<br/>FALLBACK_USED=1"]
    escalate -- "no" --> fallback
    fallback --> ok4{validated?}
    ok4 -- "yes" --> apply_fb([apply RESPONSE<br/><b>NOT cached under primary key</b>])
    ok4 -- "no" --> abort_explicit([echo &quot;[clio] step &lt;name&gt;: msg&quot;<br/>exit 1])

    classDef terminal fill:#fee,stroke:#c33
    classDef success fill:#efe,stroke:#3c3
    class abort_explicit terminal
    class apply,apply_fb success
```

If the chain doesn't end with an explicit `abort`, the runtime emits an implicit abort with `[clio] step <name>: ON_FAIL strategies exhausted` whenever `RESPONSE_NN` is still empty.

### Implicit abort

When all clauses are exhausted and none ended with `abort`, the runtime emits an implicit abort: exit 1 with `[clio] step <name>: ON_FAIL strategies exhausted`.

### Compatibility checks (`fallback`)

At IR build time, for every `fallback(step_name)` reference:

1. The named step must exist (otherwise `IRBuildError`).
2. Its `MODE` must be either `exact` or `judgment` (any mode is allowed — the rule-based exact fallback is a common pattern).
3. Its `takes` must be **structurally equal** to the parent step's `takes` (same field names in the same order, types match per the existing `types_equal` helper).
4. Its `gives` must be structurally equal to the parent step's `gives`.
5. Cycles in the fallback graph are rejected (`step a → fallback b → fallback a`). Detected by DFS at IR build time.

These checks produce the standard `IRBuildError` with `<file>:<line>:<col>: ...` formatting.

### Restriction

ON_FAIL on `MODE: exact` is rejected at parse time (different failure semantics; deferred). The compiler error message states this explicitly.

## 6. Cache + ON_FAIL — interaction

- Cache lookup happens **once**, at the very beginning of the step, before any LLM call. Hit → use cached response, skip the strategy chain entirely.
- On miss, the strategy chain runs. The cache key is recomputed for `escalate` (different `model` → different key). Each model's successful response is cached under its own key.
- Successful response from a `fallback(step)` is NOT cached under the main step's key (the fallback manages its own cache).
- Therefore, after a flow run that ended via `fallback`, the main step's cache is still empty; the next run will retry the main step's chain from scratch (and possibly re-fall-back). This is the intended behavior — caching a fallback under the main step would mask the underlying transient failure permanently.

## 8. Run.sh shape — emitted bash for a judgment step with CACHE+ON_FAIL

The emitter defines a bash helper at the top of `run.sh` to keep step bodies readable. Helper signature:

```bash
_clio_run_attempt() {
    # args: $1=model $2=prompt $3=schema_path $4=schema_inlined
    # stdout: cleaned response on success, empty on failure
    # exit code: 0 on success+validated, 1 on any failure
    local model="$1" prompt="$2" schema_path="$3" schema_inlined="$4"
    local raw clean
    raw="$(printf %s "$prompt" | claude -p --model "$model" --output-format text 2>/dev/null || true)"
    [ -n "$raw" ] || return 1
    clean="$(printf %s "$raw" | awk '!/^```/')"
    printf %s "$clean" | "$PYTHON" -m clio_runtime.validate "$schema_path" - >/dev/null 2>&1 || return 1
    printf %s "$clean"
    return 0
}
```

Per-step emitted code (illustrative, exact form is the golden truth):

```bash
# Step 2: detect_churn (judgment, cache=ttl(86400), on_fail=retry(3) then escalate then fallback(detect_churn_naive) then abort)
INLINED_SCHEMA_02='{"type":"array",...}'
PROMPT_02="$("$PYTHON" -m clio_runtime.substitute steps/02_detect_churn.prompt state.json)"
PROMPT_02="${PROMPT_02//\$\{schema\}/$INLINED_SCHEMA_02}"

CACHE_DIR_02="${CLIO_CACHE_DIR:-.cache}"
MODELS_02=(haiku sonnet opus)
MODEL_IDX_02=0
RESPONSE_02=""

# Cache lookup against the first model
KEY_02="$("$PYTHON" -m clio_runtime.cache key detect_churn "${MODELS_02[0]}" "$PROMPT_02" "$INLINED_SCHEMA_02")"
RESPONSE_02="$("$PYTHON" -m clio_runtime.cache lookup "$CACHE_DIR_02" detect_churn "$KEY_02" 86400 2>/dev/null || true)"

if [ -z "$RESPONSE_02" ]; then
    # Strategy 1: retry(3) on current model — i.e. up to 4 attempts total
    for attempt_02 in 1 2 3 4; do
        if RESPONSE_02="$(_clio_run_attempt "${MODELS_02[$MODEL_IDX_02]}" "$PROMPT_02" steps/02_detect_churn.schema.json "$INLINED_SCHEMA_02")"; then
            break
        fi
        RESPONSE_02=""
    done

    # Strategy 2: escalate — bump model and one attempt
    if [ -z "$RESPONSE_02" ] && [ $MODEL_IDX_02 -lt $((${#MODELS_02[@]} - 1)) ]; then
        MODEL_IDX_02=$((MODEL_IDX_02 + 1))
        # Recompute cache key for the new model and check that too
        KEY_02_ESC="$("$PYTHON" -m clio_runtime.cache key detect_churn "${MODELS_02[$MODEL_IDX_02]}" "$PROMPT_02" "$INLINED_SCHEMA_02")"
        RESPONSE_02="$("$PYTHON" -m clio_runtime.cache lookup "$CACHE_DIR_02" detect_churn "$KEY_02_ESC" 86400 2>/dev/null || true)"
        if [ -z "$RESPONSE_02" ]; then
            RESPONSE_02="$(_clio_run_attempt "${MODELS_02[$MODEL_IDX_02]}" "$PROMPT_02" steps/02_detect_churn.schema.json "$INLINED_SCHEMA_02" || true)"
        fi
        # If we got a response, also store under this model's key
        if [ -n "$RESPONSE_02" ]; then
            "$PYTHON" -m clio_runtime.cache store "$CACHE_DIR_02" detect_churn "$KEY_02_ESC" "${MODELS_02[$MODEL_IDX_02]}" "$RESPONSE_02"
        fi
    fi

    # Strategy 3: fallback(detect_churn_naive)
    if [ -z "$RESPONSE_02" ]; then
        # Run the fallback step (own bash block, generated like any other exact/judgment step).
        # It reads its inputs from state.json same as the main step would.
        "$PYTHON" steps/03_detect_churn_naive.py --customers="$(jq -r .customers state.json)"
        RESPONSE_02="$(jq -c .risks state.json)"
        # Fallback success is NOT cached under detect_churn's key.
    fi

    # Strategy 4: abort("...")
    if [ -z "$RESPONSE_02" ]; then
        echo "[clio] step detect_churn: churn detection exhausted" >&2
        exit 1
    fi

    # Cache successful main-step response (only when strategies 1 or 2 won)
    if [ $MODEL_IDX_02 -eq 0 ] && [ -n "$RESPONSE_02" ]; then
        "$PYTHON" -m clio_runtime.cache store "$CACHE_DIR_02" detect_churn "$KEY_02" "${MODELS_02[0]}" "$RESPONSE_02"
    fi
fi

# Apply: write the validated response into state.json under the GIVES name
jq --argjson r "$RESPONSE_02" '.risks = $r' state.json > state.json.tmp && mv state.json.tmp state.json
```

Verbose by design — every strategy is observable in the emitted code. No hidden orchestrator logic in clio_runtime.

## 11. CLI surface

Unchanged. `python -m clio compile` and `python -m clio check`.

The emitted project gains one runtime CLI tool: `python -m clio_runtime.cache <subcommand>`. This is internal — no user is expected to call it manually, but it's documented in the module docstring.

## 12. Open points (acknowledged, not blocking v0.2)

- **Cache invalidation on schema change**: covered by including the schema in the cache key. If a contract changes, all dependent step caches naturally miss. No explicit invalidation needed.
- **Concurrent writers to the same cache file**: handled by atomic `mv` from `.tmp`. Two parallel runs writing the same key result in one of the two persisting (last writer wins, but no corrupted file).
- **Cache for fallback step itself**: the fallback step caches independently if it has its own `CACHE` field. v0.2 does not propagate caching policy from main → fallback.
- **Recompiling does NOT clear cache**: the compiler is stateless. The output dir is the cache root. Re-compiling overwrites step files but leaves `.cache/` alone. To clear, `rm -rf .cache/` or set `CLIO_CACHE_DIR` to a fresh path.

## 13. What this design rules out

- **No silent caching of unvalidated responses**. A response that fails the schema or `x-clio-assert` is never written to cache.
- **No silent silent fallthrough on unknown ON_FAIL strategies**. Parser is the single source of truth for the strategy enum.
- **No cross-step cache key collision**. The step name is part of the key.
- **No automatic retry backoff in v0.2**. Retries are immediate. (Future: `retry(3, backoff="exponential")` if needed.)
